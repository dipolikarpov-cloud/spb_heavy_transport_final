## **Описание выбранных методологий обработки данных**

Первой задачей было выгрузить данные с “Система классификаторов и открытых данных Санкт-Петербурга”. Самым проблемным слоем был Технико-экономические паспорта многоквартирных домов (Версия №11 от 17.03.2026) так как у каждого дома были указаны адреса, но не было координат, что ставило перед нами задачу геокодировать 22 тысячи единиц зданий. Уникальность этого слоя состояла в том, что у каждого дома указано число проживающих, что позволяет повысить точность расчетов, не прибегая к расчету по нормативам. Специально под эту задачу был написан соответствующий скрипт на Python с использованием библиотеки geopy и геокодера Nominatim (OpenStreetMap). Код был сгенерирован с помощью нейросети Claude (Anthropic).  
Исходный файл содержал 22 577 записей с адресами в формате "улица, дом, корпус, литера" без координат. Задача состояла в том, чтобы преобразовать текстовые адреса в географические координаты (широта и долгота).  
Скрипт выполнял следующие операции: предварительную очистку адресов  (удаление лишних элементов ("дом", "литера"), замену "корпус" на "к", раскрытие аббревиатуры "В.О." до "Васильевского острова)"; последовательную отправку запросов к API Nominatim с задержкой 1 секунда между запросами во избежание блокировки; сохранение промежуточных результатов каждые 50 записей для защиты от потери данных при обрывах соединения; возможность возобновления работы с места остановки при повторном запуске.В результате нескольких итераций геокодирования удалось получить координаты для 20 276 зданий из 22 577, что составило около 90% от исходного массива данных. Оставшиеся записи либо относились к нестандартным адресам, либо отсутствовали в базе OpenStreetMap. Полученное покрытие в 90% было признано достаточным для целей настоящей карты: поскольку визуализация основана не на отдельных домах, а на агрегированных значениях в ячейках гексагональной сетки, где каждая ячейка может включать десятки и сотни объектов, отсутствие 10% записей не оказывает существенного влияния на итоговую картину распределения населения по территории города.  
Следующей задачей было построить изохроны от жд станций и станций метро. На этом этапе мы столкнулись с проблемами выгрузки улично-дорожной сети через QuickOSM и Overpass Turbo. При изучении иных методов были принято решение написать скрипт, использующий библиотеки osmnx и networkx. Преимущества этого метода сводиться к тому, что библиотека osmnx выгружает уже размеченную улично-дорожную сеть OpenStreetMap, где каждый перекресток отображается вершиной, а дороги между ними ребрами. Этот формат является крайне удобным для библиотеки networkx, расчитывающая изохроны при помощи алгоритма Дейкстры. Также это позволяет избавляет нас от необходимости вручную фильтровать данные линейных слоев улично-дорожной сети в QGIS, так как при выгрузке графа в osmnx можно указать параметр network\_type со значением “walk”, тогда выгружается пешеходный граф территории. Изохроны считались в 15 минут так как в местной градостроительной документации не указаны нормативы доступности для этих объектов.  
Следующий этап сводился к расчету количества жителей живущих в пешей доступности от каждой станции метро и жд станции. Для этого использовался готовый слой изохрон к которому при помощи “Join Attributes by Location” из слоя с домами. Затем эти данные были перенесены в атрибутивные таблицы станций метро и жд станций.  
Наконец, данные были перенесены в GitHub. Проект был экспортирован из QGIS с помощью плагина qgis2web в формат веб\-карты на базе библиотеки Leaflet. В результате экспорта была сформирована папка проекта, включающая файл index.html, папку data с пространственными слоями в формате GeoJSON/JavaScript, а также папки css, js, legend и markers, содержащие стили, скрипты, элементы легенды и SVG-маркеры. Полученная структура была загружена в репозиторий GitHub и опубликована через GitHub Pages. После публикации была выполнена ручная доработка файла index.html: исправлены ошибки экспорта qgis2web, настроены названия слоев в легенде, очищены pop-up окна, скорректирована символизация отдельных точечных слоев, добавлена информационная плашка с описанием назначения карты и методологии.

## **Что было самым сложным в работе?**

Из основных сложностей работы можно выделить два основных элемента работы: геокодирование домов Санкт-Петербурга и перенос карты в GitHub. Вот список основных трудностей, с которыми мы столкнулись:

1) Нестандартный формат адресов в исходных данных: наличие литер, корпусов и сокращений (В.О., г., пос.) потребовало многократной итеративной очистки адресной строки перед отправкой запросов к геокодеру.  
2) Ограничения API Nominatim: лимит в 1 запрос в секунду между каждым запросом в купе с неоднократной необходимостью очищать “догеокодировать” негеокодированные адреса существенно увеличили время обработки до де\-факто целого дня, что делает выбранный метод геокодирования неоптимальным при работе с большими объемами данных.   
3) Ошибка в URL базовой подложки CartoDB Positron: при экспорте из qgis2web в index.html внутри строки с URL подложки появлялся лишний перенос строки, из\-за чего карта показывала белый экран; после каждого экспорта (а их, к сожалению, было больше двух) эту строку приходилось вручную исправлять.  
4) Проблемы со сложными маркерами: часть кастомной символизации из QGIS не переносилась на веб\-карту корректно: вместо SVG-иконок для станций, парковок, депо и остановок появлялись стандартные маркеры Leaflet, лишние серые точки или слишком маленькие SVG; поэтому составные маркеры приходилось вручную вносить в index.html.   
5) Проблемы с кластеризацией остановок: при кластеризации остановок пользовательские маркеры заменялись на типовые цветные кружки, а при приближении могли появляться кластеры с “лучами” вместо нормальных одиночных остановок; поэтому отдельно настраивались цвета кластеров для автобусных, трамвайных и троллейбусных остановок, а одиночные маркеры приводились к нужному виду вручную.  
6) Слишком большой объем исходной карты: изначально планировалась единая карта по транспортной инфраструктуре, но из\-за большого количества слоев, пришлось разделить ее на две карты: по тяжелому и легкому транспорту

В данной работе мы гордимся

1) Создать веб\-карту, выгрузить ее на github, получить первый опыт создания интерактивных карт для аналитики

3) Геокодировать около 20 тысяч жилых зданий бесплатно, при помощи самостоятельно написанного скрипта при использовании нейронной сети

## **Как использовали нейросети**

1) При написании кода для геокодирования, расчета изохрон и перевода слоя с маршрутами общественного транспорта из точек в линии;  
2) Неоценимую помощь оказали нейросети Claude и ChatGPT при поиске и исправлении ошибок с html и geojson файлах. При обнаружении какой-либо ошибки на карте, нейросети отправлялся запрос с подробным описанием имеющейся проблемы и четко зафиксированным желаемым результатом, после чего нейросеть находился в огромном коде конкретную ошибку и присылала исправленный код. Нейросети не всегда удавалось исправить ошибку с первого раза, поэтому зачастую на это уходило несколько итераций.
